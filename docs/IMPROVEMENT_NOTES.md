外部ツール・手法との比較分析、運用を通じて発見された課題、未整備のテンプレート計画などを蓄積します。

## 1. 仕様駆動開発（SDD）との比較分析

### 1.1 背景

仕様駆動開発（Spec-Driven Development / SDD）は、AI駆動開発において「実装前に仕様を固め、AIと人間の認識を合わせる」ことで手戻りを防止する手法である。Web Starter Kit と SDDツール群は根底にある思想（要件定義ファースト・承認ゲート・推測禁止・セッション間の記憶永続化）が非常に近いが、アプローチの粒度と対象領域に違いがある。


### 1.2 共通する思想

| 思想 | SDDツール群 | Web Starter Kit |
|---|---|---|
| 実装前に仕様を固める | 要件→設計→タスク→実装の順を強制 | ヒアリング→要件定義→SOW→実装の順を強制（AGENTS.md §0） |
| 承認ゲート | 各フェーズで承認してから次へ進む | 要件合意→SOW合意→品質ゲート通過の多段承認 |
| AIの推測を禁止 | EARS形式で曖昧さを排除 | 「推測で要件を埋めない」「TBDはTBDのまま明示」 |
| セッション間の記憶を永続化 | Steeringドキュメント（`product.md`, `tech.md`, `structure.md`） | `AGENTS.md` + `docs/governance/rules/` 規約群 |
| タスクのトレーサビリティ | `tasks.md` + `validate-impl` | SOW受入基準 + `QUALITY_GATES.md` |

### 1.3 Web Starter Kit が優位な点

| 観点 | 詳細 |
|---|---|
| **スコープの広さ** | プロジェクト全体のライフサイクル（要件定義〜リリース〜保守）をカバー。SDDツール群は機能単位の仕様策定に特化 |
| **規約の深さ** | SEO / セキュリティ / a11y / パフォーマンス / メール / LLMO まで国際標準ベースで個別規約化。SDDツール群はコーディング規約まで |
| **品質保証の体系** | lint / typecheck / test / build / secret scan / vuln scan の自動品質ゲート + Lighthouse / Pa11y / DAST まで段階設計 |
| **非機能要件のカバー** | CSP / HSTS / CWV / フォント最適化 / DNS認証など非機能要件を体系的にカバー |
| **ツール非依存** | 特定のAIツールに依存せず、どのAIエージェントでも、AIなしでも同じ手順で運用可能 |

### 1.4 Web Starter Kit の改善余地（SDD手法から学べる点）

以下の5点は、SDD手法が仕組みとして解決しているが、Web Starter Kit では未対応または弱い領域である。

#### ① 要件の構造化フォーマットの導入 ✅ 導入済み

**現状の課題**: ヒアリングシートと要件定義書は自然言語のテーブル形式。「TBDを禁止」「推測禁止」とルールはあるが、記述フォーマット自体が曖昧さを構造的に排除する仕組みになっていない。同じ要件でもAIエージェントによって粒度がバラつく可能性がある。

**SDDツール群のアプローチ**: EARS形式（`WHEN ... THEN ...` / `IF ... THEN ...`）で条件と結果を機械的に明確化。Feature Specs では全要件をEARS記法で記述し、テスト可能性・トレーサビリティの基盤としている。

**対応内容**: `REQUIREMENTS_TEMPLATE.md` §2.5 に EARS記法の記述ガイドラインを追加。条件分岐・エラーケースを含む機能要件はEARS形式で記述することを推奨（Should）とした。

**ステータス**: ✅ 導入済み（2026-04-13）

---

#### ② 機能単位の仕様管理 ✅ 導入済み

**現状の課題**: プロジェクト全体に対して1つの要件定義書 + フェーズ別SOW。個別機能レベルの仕様分割の仕組みがない。大規模プロジェクトでは要件定義書が肥大化し「この機能の仕様はどこ？」が探しにくくなる可能性がある。

**SDDツール群のアプローチ**: 機能ごとにディレクトリを切り、requirements.md → design.md → tasks.md が1セットで管理。

**対応内容**: SDDの3ファイル構造をツール非依存な1ファイルに統合した `FEATURE_SPEC_TEMPLATE.md` を新規作成。保存先は `docs/projects/<project_slug>/specs/<feature_slug>.md`。加えて、バグ修正にも仕様化アプローチを適用する `BUGFIX_SPEC_TEMPLATE.md` を新規作成（「何を壊さないか」を明示するリグレッション防止の思想を導入）。`AGENTS.md` §0 に作成判断基準を追記済み。

**ステータス**: ✅ 導入済み（2026-04-13）

---

#### ③ 要件⇔実装の自動トレーサビリティ 🔶 部分導入済み

**現状の課題**: 品質ゲート（lint/test/build等）はコードの品質を検証するが、「要件定義書のこの項目が実装に反映されたか」という要件⇔実装のトレーサビリティを自動検証する仕組みがない。受入基準の確認は人手（またはAIの自己申告）に依存。

**SDDツール群のアプローチ**: `tasks.md` の `[ ]→[x]` 自動更新 + 検証コマンドで要件充足を機械的に検証。

**対応内容（部分導入）**: `SOW_TEMPLATE.md` §4 を「受入基準チェックリスト」形式に拡張し、要件IDと検証方法を紐づける構造にした。`FEATURE_SPEC_TEMPLATE.md` にも §5「完了確認」チェックリストを設置。ただし自動検証スクリプトとの連動は未実装（将来課題）。

**残課題**: 受入基準チェックリストとE2Eテスト/CIの自動紐づけ

**ステータス**: 🔶 部分導入済み（2026-04-13）。自動検証は将来課題。

---

#### ④ ワークフローのコマンド化・自動化

**現状の課題**: AGENTS.md に作業手順は明記されているが、ワークフローのエンフォースメント（強制力）がドキュメント記述に留まっている。テンプレートのコピー、ディレクトリ作成、品質ゲート実行などは手動。

**SDDツール群のアプローチ**: カスタムスラッシュコマンドでワークフローをコマンド駆動で実行。迷わず進められる。

**改善案**:
- `scripts/init-project.sh <project_slug>` のようなプロジェクト初期化スクリプトの提供（テンプレートコピー + ディレクトリ作成の自動化）
- `scripts/quality-gate.sh` による品質ゲート一括実行スクリプトの提供
- ただし、Agent Skills（`.gemini/` / `.agents/` 等のAIツール固有のワークフロー定義）は意図的に導入しない方針（README.md「Agent Skills について」参照）を維持する

**優先度**: Medium — プロジェクト初期化の自動化は早期に実現可能で効果が高い

---

#### ⑤ バックエンド設計成果物テンプレートの追加 🔶 部分導入済み

**現状の課題**: 「詳細設計書は作らない」方針で、UI設計はワイヤーフレーム/HTMLモックで代替。しかしバックエンド設計（APIフロー、状態遷移、シーケンス図等）の成果物テンプレートがない。SOWの「成果物」欄に自由記述で追加する前提だが、フォーマットは未定義。

**SDDツール群のアプローチ**: `design.md` に Mermaid によるシステムフロー図、アーキテクチャ決定記録が自動生成される。

**対応内容（部分導入）**: `FEATURE_SPEC_TEMPLATE.md` §3「設計」セクションにアーキテクチャ概要（Mermaid図推奨）、技術選定と根拠テーブル、エラーハンドリングテーブル、テスト戦略を組み込んだ。独立したAPI仕様/DB設計テンプレートは未作成だが、機能仕様書内で設計を記述できる構造は整った。

**残課題**: 独立したAPI仕様テンプレート、DB設計テンプレート（頻出パターンが見えた段階で追加）

**ステータス**: 🔶 部分導入済み（2026-04-13）。独立テンプレートは将来課題。

## 2. 保守運用体制テンプレートの整備

### 2.1 背景

現在の Web Starter Kit は「開発〜リリース」のプロセスは充実しているが、**リリース後の保守・運用フェーズ**を体系的にカバーするテンプレートが不足している。クライアント納品案件や長期運用プロジェクトでは、保守体制・障害対応・SLA合意が不可欠であり、これらのテンプレートを標準化することで運用品質を底上げできる。

### 2.2 整備予定テンプレート

#### ① 保守体制・責任分担表（RACI）テンプレート

**目的**: 保守フェーズにおける「誰が何に責任を持つか」を明確化する。

**内容**:
- Responsible（実行責任者）/ Accountable（説明責任者）/ Consulted（相談先）/ Informed（報告先）のマトリクス
- 対象領域: セキュリティパッチ適用、依存更新、バックアップ、監視対応、コンテンツ更新 等

**優先度**: Medium — クライアント案件で保守契約を締結する際に必要

---

#### ② 障害対応ランブック（インシデント対応・復旧手順）

**目的**: 障害発生時に迅速かつ一貫した対応を行うための手順書テンプレート。

**内容**:
- インシデント検知 → 影響範囲判定 → エスカレーション → 復旧 → ポストモーテムの標準フロー
- 重大度レベル（Sev1〜Sev3）の定義と対応SLA
- 連絡先リスト、ステータスページ更新手順
- ロールバック手順（デプロイ先別）

**優先度**: Medium — 本番運用開始後に必要。`docs/governance/runbooks/INCIDENT_CHECKLIST.md` として配置予定（`QUALITY_GATES.md` §4 で既に言及あり）

---

#### ③ SLA/SLO 定義テンプレート

**目的**: クライアントとの稼働率目標・応答時間の合意書テンプレート。

**内容**:
- SLA（Service Level Agreement）: 稼働率保証、障害対応時間、メンテナンス窓の定義
- SLO（Service Level Objective）: 内部目標（SLAより厳しい値を設定）
- SLI（Service Level Indicator）: 計測指標の定義（可用性、レイテンシ、エラー率）
- ペナルティ/エスカレーション条件

**優先度**: Low — 大規模案件やSaaS型プロダクト向け。小規模コーポレートサイトでは過剰な場合あり

---

#### ④ 定期運用チェックリスト

**目的**: 月次・四半期ごとの定期点検項目を標準化する。

**内容**:
- 依存パッケージの脆弱性・更新確認（`OPERATIONS.md` §4.1 と連動）
- バックアップの復旧テスト（実際にリストアできるか検証）
- SSL証明書の有効期限確認
- DNS設定の整合性確認
- ログ・監視アラートの精度確認（ノイズ除去）
- CMS / 外部サービスの利用状況・コスト確認

**優先度**: Medium — 保守契約がある案件では早期に整備すべき

## 3. GitHub コミュニティスタンダード準拠テンプレートの整備

### 3.1 背景

GitHub はリポジトリの「コミュニティスタンダード」として、OSS プロジェクトやチーム開発に必要なドキュメント群を推奨している。現在の Web Starter Kit には `LICENSE` と `README.md` は存在するが、コントリビューションガイドや Issue/PR テンプレートは未整備である。チーム開発や OSS 公開を視野に入れる場合、これらの整備が有効。

参考: [GitHub Community Standards](https://docs.github.com/en/communities/setting-up-your-project-for-healthy-contributions)

### 3.2 整備予定テンプレート

#### ① `CONTRIBUTING.md`（コントリビューションガイドライン）

**内容**:
- 開発環境のセットアップ手順（Docker前提）
- ブランチ命名規則、PR作成のフロー
- コミットメッセージ規約（`DEV_RULES.md` §3.1 Conventional Commits への参照）
- 品質ゲート通過の必須条件（`QUALITY_GATES.md` への参照）
- コードレビューのガイドライン

**優先度**: Medium — チーム開発を開始する際に必要

---

#### ② `.github/ISSUE_TEMPLATE/`（Issue テンプレート）

**内容**:
- バグ報告テンプレート（再現手順、期待される動作、環境情報）
- 機能要望テンプレート（課題の背景、提案する解決策、代替案）
- タスクテンプレート（対応内容、受入基準、関連ドキュメント）

**優先度**: Medium — Issue駆動で開発を管理する場合に有効

---

#### ③ `.github/PULL_REQUEST_TEMPLATE.md`（PR テンプレート）

**内容**:
- 変更概要、関連 Issue、変更種別（feat/fix/refactor 等）
- 品質ゲート通過の自己チェックリスト
- スクリーンショット（UI変更時）
- レビュー観点の明示

**優先度**: Medium — PR ベースのレビューフローを導入する際に必要

---

#### ④ `CODE_OF_CONDUCT.md`（行動規範）

**内容**:
- [Contributor Covenant](https://www.contributor-covenant.org/)（業界標準の行動規範）の採用を推奨
- 日本語版も提供されている

**優先度**: Low — OSS 公開時に必要。社内プロジェクトでは任意

---

### 3.3 導入判断

これらのテンプレートを導入するかは、`HEARING_SHEET.md` §10「GitHub コミュニティスタンダード準拠の要否」でプロジェクトごとに合意する（既にヒアリング項目として定義済み）。

## 4. 総括

Web Starter Kit の強みは「プロジェクト全体のガバナンスを国際標準ベースで体系化している」点にある。改善余地は大きく3つのカテゴリに分類される:

1. **仕様駆動開発（§1）**: 「ルールの強制力」と「機能単位の仕様管理」の仕組み化
2. **保守運用体制（§2）**: リリース後の運用品質を担保するテンプレート群
3. **コミュニティスタンダード（§3）**: チーム開発・OSS対応のドキュメント群

改善の方向性としては、特定ツール依存のアプローチではなく、**テンプレートとスクリプトによる軽量な仕組み化**が本プロジェクトの設計思想（ツール非依存・低ロックイン）と整合する。

すべてを一度に導入する必要はなく、実運用で課題が顕在化した項目から段階的に取り組むことを推奨する。

## 5. spec-kit（仕様駆動開発ツール）との連携

### 5.1 背景

GitHub [spec-kit](https://github.com/github/spec-kit) は仕様駆動開発（Spec-Driven Development）のツールキットで、AI を活用した仕様策定→計画→タスク分解→実装のパイプラインを提供する。Web Starter Kit との比較分析を行った結果、**spec-kit の機能を WSK に取り込む**のではなく、**spec-kit のワークフロー上で WSK の規約群を活用する**方針を採用した。

### 5.2 方針の根拠

| 観点 | Web Starter Kit | spec-kit |
|---|---|---|
| **役割** | プロジェクトのガバナンス（規約・品質基準・テンプレート） | 仕様→実装パイプラインの自動化 |
| **スコープ** | プロジェクト全体のライフサイクル（8フェーズ） | 機能単位の仕様→実装（5フェーズ） |
| **品質保証** | 7項目必須ゲート + Lighthouse/Pa11y/DAST | Constitution Check |
| **ツール依存** | なし（AI不要でも運用可能） | Python/uv + 特定AIエージェント |

**結論**: 両者は「規約の定義」と「ワークフローの実行」という異なるレイヤーを担当しており、競合しない。spec-kit の Constitution に WSK の規約群を読み込ませることで、spec-kit が生成する仕様・タスクに WSK の品質基準が適用される。

### 5.3 規約群の独立リポジトリ化 ✅ 実施済み

spec-kit プロジェクトに WSK 全体をサブモジュールとして取り込むと、AGENTS.md / CLAUDE.md が spec-kit 側と衝突する問題が発見された。これを解決するため、規約群を独立リポジトリに分離した。

**リポジトリ**: [murakami-shoten/nextjs-web-governance](https://github.com/murakami-shoten/nextjs-web-governance)

```
nextjs-web-governance/
├── rules/           ← 13ファイル（開発・アーキテクチャ・セキュリティ等）
├── requirements/    ← 8テンプレート（ヒアリングシート・要件定義・SOW等）
├── runbooks/        ← RELEASE_CHECKLIST.md
├── README.md
└── LICENSE
```

- WSK では `docs/governance/` にサブモジュールとして参照
- spec-kit プロジェクトでも `docs/governance/` に同じサブモジュールを追加（AGENTS.md/CLAUDE.md の衝突なし）

**ステータス**: ✅ 実施済み（2026-04-14）

### 5.4 spec-kit 連携インストーラー ✅ 作成済み

**対応ファイル**: `scripts/install-for-speckit.sh`

spec-kit プロジェクトに nextjs-web-governance をサブモジュールとして導入するワンコマンドインストーラー。以下を自動化する:

1. `docs/governance/` に nextjs-web-governance を Git サブモジュールとして追加
2. `.specify/memory/constitution-prompt.md` を配置（Constitution 生成用プロンプト）

Constitution 自体は spec-kit の正規フロー（`/speckit.constitution` コマンド）で AI が生成する。

**利用手順**: `README.md`「spec-kit（仕様駆動開発ツール）との連携」セクション参照。

**ステータス**: ✅ 作成済み（2026-04-14）

### 5.5 テンプレート改善（spec-kit の良い点の取り込み） ✅ 導入済み

spec-kit の仕様テンプレートから有用な概念を3点抽出し、WSK のテンプレートに直接組み込んだ（spec-kit のインストール不要で効果を得られる）:

| 改善 | 対応ファイル | 内容 |
|---|---|---|
| **A. `[NEEDS CLARIFICATION]` マーカー** | `FEATURE_SPEC_TEMPLATE.md`, `BUGFIX_SPEC_TEMPLATE.md` | AI が推測で値を埋めることを構造的に防止 |
| **B. Given/When/Then 受入シナリオ** | `FEATURE_SPEC_TEMPLATE.md` §2.2b | BDD 形式でシナリオを記述 |
| **C. Phase -1 Gate** | `FEATURE_SPEC_TEMPLATE.md` §3.0 | 過剰設計防止の5項目チェックリスト |

**ステータス**: ✅ 導入済み（2026-04-14）

### 5.6 Extension/Preset による自動配布の見送り

spec-kit の拡張機構（Extension / Preset）は **コマンドとテンプレートのみ** をインストールでき、規約ファイル群の自動配布には対応していない。そのため Git サブモジュール方式を採用した。将来的に spec-kit に `post_install` フック等が追加された場合、再検討する。
