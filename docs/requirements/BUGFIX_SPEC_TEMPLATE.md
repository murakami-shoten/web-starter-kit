# バグ修正仕様書テンプレート（Bugfix Spec）

> 複雑なバグの根本原因分析・修正設計・リグレッション防止を構造化するテンプレート。
> 複製して `docs/requirements/projects/<project_slug>/specs/bugfix-<issue_slug>.md` に保存する（テンプレは編集しない）。

**全バグに作成する必要はない。** 以下のいずれかに該当する場合にのみ使用する:
- クリティカルパス（CV導線・決済・認証等）のバグ
- 過去の修正でリグレッションが発生した箇所
- 根本原因がすぐに特定できない複雑なバグ
- コンプライアンス/セキュリティに関わるバグ

該当しないバグ（タイポ、単純なロジックエラー、1行修正で解決するもの）は即席修正で十分。

参考: 仕様駆動開発（SDD）における Bugfix Spec の「何を直すか」だけでなく「何を壊さないか」を明示する思想を、ツール非依存な形式に翻訳したもの。

### `[NEEDS CLARIFICATION]` マーカーについて

根本原因が不明な段階では、推測で原因を記述せず `[NEEDS CLARIFICATION: <理由>]` マーカーを使用する。詳細は `FEATURE_SPEC_TEMPLATE.md` のマーカーガイドを参照。

## 2. 動作分析（EARS形式）

*EARS記法で「現在」「期待」「不変」の3つの動作を明示する。特に §2.3 が Bugfix Spec の核心。*

### 2.1 現在の動作（不具合）

- WHEN [条件] THEN the system [不正な動作を記述]

### 2.2 期待される動作（修正後）

- WHEN [条件] THE SYSTEM SHALL [正しい動作を記述]

### 2.3 変更してはならない動作（リグレッション防止）

*修正によって影響を受ける可能性があるが、動作を維持すべき箇所を列挙する。*

- WHEN [条件] THE SYSTEM SHALL CONTINUE TO [既存の正常動作1]
- WHEN [条件] THE SYSTEM SHALL CONTINUE TO [既存の正常動作2]
