## 2.5 機能要件の記述ガイドライン（EARS記法）

*条件分岐・エラーケース・ユーザー操作を含む機能要件は、以下のEARS（Easy Approach to Requirements Syntax）記法で記述することを推奨する（Should）。静的コンテンツ要件やデザイン要件、NFRの目標値記述など、条件分岐を含まない要件は自然言語テーブルで十分。*

**基本パターン:**

- **イベント駆動**: `WHEN [条件/イベント] THE SYSTEM SHALL [期待される動作]`
- **条件付き**: `IF [前提条件], WHEN [イベント] THE SYSTEM SHALL [動作]`
- **否定（不正ケース）**: `WHEN [不正入力/異常条件] THE SYSTEM SHALL [エラー処理]`

**記述例:**

| ID | 要件（EARS形式） | Must/Should |
|---|---|---|
| FR-001 | WHEN ユーザーが問い合わせフォームを送信する THE SYSTEM SHALL 入力データを検証し、管理者にメール通知する | Must |
| FR-002 | IF 入力にバリデーションエラーがある, WHEN フォームを送信する THE SYSTEM SHALL 該当フィールドの横にエラーメッセージを表示する | Must |
| FR-003 | WHEN reCAPTCHA検証に失敗する THE SYSTEM SHALL フォーム送信をブロックし、リトライを促すメッセージを表示する | Should |

> **EARS形式を使うべきケース**: 条件分岐、エラー処理、ユーザー操作フロー、状態遷移
> **自然言語で十分なケース**: 静的コンテンツ要件、デザイン要件、NFRの目標値記述
